Compound Drawer API: Two Approaches

How should DrawerPage children declare their slots (title, bottomPanel, etc.)?

A

Slot Components (Headless-style)

Children use <DrawerTitle>, <DrawerBody>, <DrawerBottomPanel> to declare content that renders into the correct physical location via React context portals.

<Drawer pagination={pagination}>
  <DrawerPage id="details">
    <DrawerTitle>Transaction Details</DrawerTitle>
    <TransactionSummary />
    <DrawerBottomPanel>
      <Button onClick={() => open('edit')}>Edit</Button>
    </DrawerBottomPanel>
  </DrawerPage>

  <DrawerPage id="edit">
    <DrawerTitle>Edit Transaction</DrawerTitle>
    <EditForm />  {/* can use DrawerBottomPanel inside */}
  </DrawerPage>
</Drawer>

Pros

  • Most flexible — slot components work at any nesting depth
  • Fully co-located: page content + its chrome live together
  • Sub-components (EditForm) can inject bottomPanel without prop drilling
  • Familiar pattern (Radix, Headless UI)

Cons

  • More internal complexity (context-based portals)
  • Need to handle multiple DrawerTitle/DrawerBottomPanel conflicts
  • Timing: portals must sync with page transitions
B

Props on DrawerPage

Each <DrawerPage> accepts title, bottomPanel, additionalActions as props. Drawer extracts them during render. A separate <DrawerBottomPanelPortal> handles the deep-nesting case.

<Drawer pagination={pagination}>
  <DrawerPage
    id="details"
    title="Transaction Details"
    bottomPanel={<Button>Edit</Button>}
  >
    <TransactionSummary />
  </DrawerPage>

  <DrawerPage id="edit" title="Edit Transaction">
    <EditForm />
    {/* Deep nesting: */}
    <DrawerBottomPanelPortal>
      <Button>Save</Button>
    </DrawerBottomPanelPortal>
  </DrawerPage>
</Drawer>

Pros

  • Simpler mental model — config is on the page, not scattered as children
  • Easy to scan: all page config visible at the DrawerPage level
  • Less internal machinery needed
  • Portal component handles the deep-nesting escape hatch

Cons

  • Props can get verbose for complex bottomPanels
  • Two patterns to learn (props for simple, Portal for deep)
  • Less "headless" — not as composable at arbitrary depth

Context Hooks (included in both approaches)

// Any child at any depth:
const { close } = useDrawer();           // close the drawer
const { open, back, currentPage } = useDrawerPagination(); // navigate pages

Both approaches include useDrawer() and useDrawerPagination() hooks to eliminate prop drilling of onClose and pagination state.
